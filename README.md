# Оптимізація часу запуску Android-додатку (Startup Time Demo)

Цей проєкт є навчальним прикладом для демонстрації різниці у швидкості запуску мобільного додатка в стані **Cold Start** (холодний запуск) та методів його оптимізації.

## 📱 Про додаток

Додаток складається з одного екрана (`MainActivity`), розробленого на **Jetpack Compose**. Для імітації реального навантаження у додатку реалізовано:
* **Синхронну затримку** в класі `Application` (імітація важких SDK або ініціалізації БД).
* **Інтерактивний інтерфейс**:
    * `TopAppBar` із заголовком.
    * `Welcome Card`: інформаційна картка з описом.
    * `Counter Card`: інтерактивний лічильник, що змінює стан при натисканні.
    * `LazyColumn`: вертикальний список із 15 елементів для демонстрації продуктивності рендерингу.

## 🚀 Методи оптимізації

У проєкті реалізовано два ключові підходи до покращення швидкості запуску:

### 1. Асинхронна ініціалізація (Coroutines)
Замість блокування головного потоку (`Main Thread`) під час `onCreate` у класі `Application`, важкі процеси перенесені у фоновий потік за допомогою `CoroutineScope(Dispatchers.Default)`. Це дозволяє системі почати малювати UI миттєво, не чекаючи завершення фонових завдань.

### 2. Baseline Profiles
Використання бібліотеки **Baseline Profile Generator** дозволяє створити список критично важливих класів та методів, які використовуються під час запуску.
* **Як це працює**: Android Runtime (ART) заздалегідь компілює ці методи в машиний код (AOT - Ahead-of-Time compilation) під час встановлення додатка.
* **Результат**: Додаток не витрачає ресурси CPU на інтерпретацію коду (JIT) під час першого запуску, що зменшує час до появи першого кадру.

## 📊 Вимірювання (Benchmarking)

Для об'єктивної оцінки результатів у модулі `:baselineprofile` додано макробенчмарки (`StartupBenchmark`), які порівнюють два стани:
1.  **CompilationMode.None**: Запуск без жодної оптимізації (чистий JIT).
2.  **CompilationMode.Partial**: Запуск із застосуванням Baseline Profile.

> При запуску тестів, результати (медіанний час запуску в мс) будуть виведені в консоль `Benchmark` після завершення 5 ітерацій кожного тесту.
